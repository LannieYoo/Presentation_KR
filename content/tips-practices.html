What Students Find Hard

  • Thinking about the world as snapshots (states) instead of step-by-step instructions
  • Remembering all the rules for when actions are allowed
  • Understanding why the computer cannot find a plan
  • Finding mistakes in Prolog code when things go wrong

Helpful Tips

  • Start small - Use just 2 places instead of 5, or 2 objects instead of 10. This makes it easier to understand and test your code. Once it works with a small problem, you can add more complexity later.
  • Draw pictures - Draw each situation on paper with arrows showing what actions change. Visualizing the states helps you see what needs to be true before an action and what changes after. This makes writing the Prolog rules much clearer.
  • Write things down first - List all objects, places, and facts before writing code. Make a simple table: Objects | Locations | Facts | Actions. This helps you organize your thoughts and avoid missing important parts.
  • Think step by step - For each action, ask: What must be true before? What changes after? What stays the same? If you can explain your action in simple words like a story, it will be easier to write the Prolog code correctly.
  • Test one thing at a time - Test single actions first, like checking if the monkey can move, before testing the whole plan. Check the starting situation, then test if actions are allowed, then see what changes. This helps you find mistakes quickly.
  • Use good names - Write at(robot, l1) instead of short names like a(r, 1). Clear names make your code easier to read and understand later. When you come back to your code after a few days, good names will help you remember what everything means.
  • Check systematically when debugging - If no plan is found, check in order: starting situation → goal condition → action rules → what changes after actions. Following this order helps you find the problem faster instead of checking randomly.

Common Mistakes

Forgetting "NOT" Rules

Some actions need something to be NOT true.

  • Example: The monkey can only move if it is not on the box
  • Problem: Many students forget to check negative conditions
  • Result: The planner fails or finds wrong plans
  • Solution: Always check both what must be true AND what must be false
