What is Knowledge Representation

Basic Idea

Knowledge Representation is a way to describe facts about the world so that a computer can reason logically about them.

We Use:

  • Objects: agents, objects, locations (e.g., robot, items, rooms)
  • Relations and Properties:
    • at(object, location) - where something is
    • has(agent, item) - what an agent holds
    • on(object1, object2) - spatial relationships
  • States: Snapshots of what is true now, described by fluents (facts that can change over time)

Good Knowledge Representation makes it clear:

  • What is true in the current state
  • Which actions are allowed
  • What the world looks like after an action

What is a Planning Problem

Planning problem in Artificial Intelligence:

  • Given an initial state
  • Given a goal condition
  • Find a sequence of actions that leads from start to goal

Examples:

  • Robot moving through rooms
  • Blocks World
  • Navigation problems

Key Elements:

  • State
  • Actions with preconditions and effects
  • Transitions between states
  • Plan as a sequence of actions

Prolog Solutions

Speaker: Peng Wang

Big Picture

In Prolog the planning system is built from:

  • State representation
  • Action representation
  • Preconditions
  • Effects
  • And a planner predicate that uses all of them together

How to Show States

A state is shown using a situation variable S and facts that describe what is true in S.

Common facts:

  • Robot is at some location
  • Locations are connected

Starting state:

A special situation, often called s0, lists what is true at the beginning.

Goal condition:

A fact or a set of facts that must be true at the end. For Simple Navigation: robot is at the target location.

Understanding Facts:

Facts are things that can change over time. They describe what is true in some states but not in others. Examples include at(robot, l1) (robot is at location 1).

Actions in Prolog

Each action is written as a Prolog term.

Example actions:

  • move(From, To) - robot moves from one location to another

For each action we later write:

  • When it is allowed (what must be true before)
  • How it changes the state (what becomes true after)

Key Planning Rules

Precondition Rule

Checks when an action is allowed in a state

poss(Action, S)

Returns true if the action is possible in state S

Effect Rule

Shows how an action changes one state into another

result(Action, S1, S2)

S2 is the new state after doing Action in state S1

Planner Rule

Finds which list of actions is a plan from a state to a goal

plan(S, G, Plan)

Plan is a list of actions that go from state S to goal G

Holds Rule

Checks if a fact (goal) is true in a given state

holds(Fluent, S)

Used to check if goal G is already true in state S

How Planning Uses Repetition

Planning Example:

% Base case: goal already satisfied
plan(S, G, []) :-
    holds(G, S).

% Recursive case: find action and continue planning
plan(S, G, [A|Plan]) :-
    poss(A, S),
    result(A, S, S1),
    plan(S1, G, Plan).

How This Works:

  • Base case: If goal G is already true in state S, the plan is empty []
  • Recursive case: Choose an action A that is possible in S, find the next state S1, then find a plan from S1 to G, and put A together with the rest of the plan

How Prolog Searches for a Plan

Prolog tries one path at a time, and if it fails, it tries a different path.

  • If the goal is already true in the current state → the plan is empty
  • Otherwise → choose an action that is possible in this state, find the next state, then continue planning from that next state
  • If this path does not work, Prolog goes back and tries another action or a different sequence

Tips and Best Practices

Speaker: Hye Ran Yoo

What Students Find Hard

  • Thinking about the world as snapshots (states) instead of step-by-step instructions
  • Remembering all the rules for when actions are allowed
  • Understanding why the computer cannot find a plan
  • Finding mistakes in Prolog code when things go wrong

Helpful Tips

  • Start small - Use just 2 places instead of 5, or 2 objects instead of 10. This makes it easier to understand and test your code. Once it works with a small problem, you can add more complexity later.
  • Draw pictures - Draw each situation on paper with arrows showing what actions change. Visualizing the states helps you see what needs to be true before an action and what changes after. This makes writing the Prolog rules much clearer.
  • Write things down first - List all objects, places, and facts before writing code. Make a simple table: Objects | Locations | Facts | Actions. This helps you organize your thoughts and avoid missing important parts.
  • Think step by step - For each action, ask: What must be true before? What changes after? What stays the same? If you can explain your action in simple words like a story, it will be easier to write the Prolog code correctly.
  • Test one thing at a time - Test single actions first, like checking if the monkey can move, before testing the whole plan. Check the starting situation, then test if actions are allowed, then see what changes. This helps you find mistakes quickly.
  • Use good names - Write at(robot, l1) instead of short names like a(r, 1). Clear names make your code easier to read and understand later. When you come back to your code after a few days, good names will help you remember what everything means.
  • Check systematically when debugging - If no plan is found, check in order: starting situation → goal condition → action rules → what changes after actions. Following this order helps you find the problem faster instead of checking randomly.

Common Mistakes

Forgetting "NOT" Rules

Some actions need something to be NOT true.

  • Example: The monkey can only move if it is not on the box
  • Problem: Many students forget to check negative conditions
  • Result: The planner fails or finds wrong plans
  • Solution: Always check both what must be true AND what must be false

Conclusion

Speaker: Jiaxing Yi

Main Point

Knowledge Representation provides a structured way to model the world using facts and logical rules, enabling systematic problem-solving through planning rather than random trial and error.

Key Takeaways

1. Knowledge Representation

Provides a way to model the world using facts and logical rules

2. Planning

Involves finding a sequence of actions from an initial state to a goal state

3. Prolog's Recursive Structure

Makes it natural for implementing planners
